# CONVENÇÕES DO BANCO DE DADOS

## ID (chave primária de todas as tabelas)
Tipo: [UUID]
Nome da coluna: id

## Campos automáticos em TODAS as tabelas
- created_at: data e hora de criação (automático)
- updated_at: data e hora da última alteração (automático)
- deleted_at: exclusão lógica (soft delete obrigatório por compliance)
- created_by: id do usuário que criou

## Datas e Horários
- Armazenamento: [TIMESTAMP com fuso horário (tz)]
- Exibição na tela: [Padrão Brasil - DD/MM/YYYY]
- Fuso horário: [America/Sao_Paulo]
- Campos de hora: [HH:MM - obrigatórios para horários de voos]

## Números e Dinheiro
- Números inteiros: [INT] ou [BIGINT] para relações transacionais
- Valores monetários: [DECIMAL(15,2)]
- Exibição de dinheiro: [R$ 1.250,90]
- Percentuais: guardar como decimal (0.10 = 10%)

## Campos de Texto
- Nomes de pessoas e empresas: [VARCHAR(150)]
- Descrições e observações: [TEXT sem limite (crucial para atendimento exceções)]
- Capitalização: [salvar como digitado]
- E-mail: VARCHAR(255), único por usuário, sempre em minúsculas
- Telefone: [VARCHAR(20) com código internacional]
- Endereço: campos separados (rua, número, bairro, cidade, state, postal_code)
- Documentos Brasileiros (obrigatórios agência de viagem):
  - CPF: VARCHAR(11) só dígitos
  - CNPJ: VARCHAR(14) só dígitos

## Relacionamentos entre tabelas
- Quando apagar registro pai: [BLOQUEAR se tiver filhos e aplicar Soft Delete]
- Nome das chaves estrangeiras: {nome_da_tabela}_id

## Nomes de tabelas e colunas
- Padrão de escrita: [snake_case]
- Idioma do código e DB: [Inglês - EN] /* Melhor adoção para GenAI */
- Tabelas no: [plural] (ex: bookings, companies, users)

## Segurança e Arquitetura B2B
- Autenticação e senha: [Supabase Auth]
- Multitenancy: Sistema Multi-empresa baseada na tabela `companies` via RLS.
- Perfis de usuário: [admin_platform, company_manager, travel_consultant, traveler]

## Status, Validações e Auditoria
- Campos de status: [texto com valores válidos CHECK (ex: pending, approved, booked, cancelled)]
- Campos obrigatórios: aplicar [NOT NULL] no banco E validação na tela.
- Histórico de Auditoria: Criar tabela de histórico de transições de status (`audit_logs` ou `{entity}_history`). Crucial para o RH do cliente.

## Performance
- Volume esperado: [médio a grande (240.000 logs anuais mínimo)]
- Índices automáticos em: todas as FKs + email + document + status + created_at
- Paginação: [20 por página para relatórios] e [Scroll infinito no Feed mobile.
